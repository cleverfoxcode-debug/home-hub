# ESP32 Framework Developer Guide

> Версия: 0.1
> Статус: Draft
> Язык: C++17
> IDE: VS Code + PlatformIO
> Framework: Arduino (с возможностью последующего перехода на ESP-IDF)

---

# Цель проекта

Создать масштабируемый и поддерживаемый фреймворк для разработки приложений под ESP32.

Проект не является набором Arduino-скетчей.

Проект представляет собой архитектурно разделенное Embedded-приложение.

---

# Основные принципы

## 1. Один класс — одна ответственность

Каждый класс выполняет только одну задачу.

Плохо:

- WiFiManager
- MQTT
- OTA
- Logger

в одном классе.

Хорошо:

Network
Logger
Storage
OTA

отдельные независимые модули.

---

## 2. main.cpp содержит минимум кода

Допустимый объем:

20–40 строк.

Пример:

```cpp
Application app;

void setup()
{
    app.begin();
}

void loop()
{
    app.update();
}
```

Никакой бизнес-логики.

---

## 3. Никаких глобальных функций

Плохо

```cpp
connectWifi();
```

Хорошо

```cpp
network.connect();
```

---

## 4. Минимум глобальных объектов

Разрешается только:

- Application
- Logger (при необходимости)

Остальные объекты принадлежат Application.

---

## 5. Никаких магических чисел

Плохо

```cpp
digitalWrite(2, HIGH);
```

Хорошо

```cpp
constexpr gpio_num_t STATUS_LED = GPIO_NUM_2;
```

или

```cpp
Pins::StatusLed
```

---

## 6. Используем современный C++

Разрешено:

- constexpr
- enum class
- inline
- std::array
- std::optional
- std::string_view
- RAII
- namespace

Запрещено:

- #define вместо constexpr
- using namespace std;
- malloc/free (если нет объективной причины)
- delay() вне специально оговоренных случаев

# Архитектура

```
Application
        │
        ├───────────────┐
        │               │
     Services       Hardware
        │               │
        │           Drivers
        │               │
        └──── ESP32 / Arduino ──── Hardware
```

Application ничего не знает о GPIO.

Application ничего не знает о UART.

Application знает только интерфейсы.

# Назначение каталогов

## app

Точка входа приложения.

Жизненный цикл программы.

---

## core

Общие интерфейсы.

Базовые классы.

Абстракции.

---

## drivers

Низкоуровневые драйверы.

Например:

- LedDriver
- ButtonDriver
- DisplayDriver

---

## hardware

Работа непосредственно с ESP32.

GPIO.

ADC.

PWM.

UART.

SPI.

I2C.

---

## services

Высокоуровневые сервисы.

Например:

- Logger
- OTA
- Network
- Storage

---

## utils

Вспомогательные классы.

Например:

- Timer
- RingBuffer
- CRC
- Helpers

# Конституция проекта

## 1. Application является корневым объектом системы

Application владеет всеми сервисами и управляет их жизненным циклом.

Никакой другой объект не создает сервисы.

---

## 2. Один владелец — один объект

Каждый сервис существует в единственном экземпляре.

Запрещается создавать сервис внутри другого сервиса.

---

## 3. Конструкторы не выполняют сложную работу

Конструктор отвечает только за создание корректного объекта.

Запрещается выполнять в конструкторе:

- подключение к Wi-Fi;
- запуск OTA;
- открытие Serial;
- создание задач;
- длительные операции.

Инициализация выполняется методом begin().

---

## 4. Единый жизненный цикл

Каждый сервис реализует интерфейс:

```cpp
begin();
update();
shutdown();
```

---

## 5. main.cpp содержит только запуск приложения

В main.cpp отсутствует бизнес-логика.

Допустимый объем:

до 40 строк.

---

## 6. Запрещены Singleton

Все зависимости должны быть явными.

---

## 7. Запрещено динамическое выделение памяти без необходимости

Не использовать:

- new
- delete

если задача решается статическим размещением объектов.

---

## 8. Каждый модуль имеет одну ответственность

Никаких "универсальных" классов.

---

## 9. Архитектурные решения документируются

Каждое важное решение фиксируется в ADR.

---

## 10. Архитектура важнее реализации

Сначала проектируется интерфейс.

Потом реализация.

Никогда наоборот.
